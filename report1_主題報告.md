## 一、專題概述（Planning）

### 1.1 動機與背景

> 在進行「服務學習」或「田野調查」等社會實踐活動時，學生常需面對長達數小時的訪談錄音。傳統的逐字稿整理極其耗時，且錄音檔往往難以即時重點標記，導致事後整理資料比較麻煩。因此，我們希望開發一個能結合即時標記與語音辨識的工具，優化從訪談到產出報告的工作流程。



### 1.2 目標使用者

> 進行田野調查與社區訪談的大學生。

  專注於質性研究的學術研究者。

  需紀錄在地故事的自媒體創作者。



### 1.3 專題目標

> 完成後，這個系統能解決什麼問題？請用條列方式列出 3–5 點。

- 目標一：提供即時標記功能，讓用戶在錄音當下能紀錄關鍵時間點。
- 目標二：整合語音辨識技術，自動將語音轉為文字
- 目標三：支援圖片與文字紀錄的時間軸同步展示。

## 二、需求分析（Requirements Analysis）

### 2.1 功能性需求（Functional Requirements）

> 系統「必須能做到」的事情。請列出核心功能（至少 3 項）。

| 功能編號 | 功能名稱 | 功能描述 |
| -------- | -------- | -------- |
| F-01     |  智能錄音與標記      |  錄音時可點擊如「重點」或「心情」等按鈕紀錄時間戳記。       |
| F-02     |  語音轉文字模組      |  將錄製好的音檔上傳至後端，產出結構化的逐字稿文字。         |
| F-03     |  內容自動排序        |  將檔案以時間順序或戳記等分類或排序                       |

### 2.2 非功能性需求（Non-functional Requirements）

> 系統的品質要求，例如效能、安全性、易用性等。

| 類別   | 需求描述 |
| ------ | -------- |
| 效能   | 語音辨識回傳時間應小於音檔長度的 50%。              |
| 安全性 | 用戶的錄音檔案需加密儲存，保護受訪者隱私。           |
| 易用性 | 介面應具備暗色模式，避免訪談時螢幕過亮干擾受訪者。    |

### 2.3 範圍限制（Out of Scope）

> 明確說明這次專題「不會」做的功能，避免範圍膨脹。

- 本次不包含：即時語音轉文字
- 本次不包含：跨裝置雲端同步
- 本次不包含：高階影音剪輯功能
